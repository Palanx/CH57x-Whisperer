# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**CH57x Whisperer** — native macOS (Apple Silicon) configurator for a CH57x-based mini keyboard: 12 keys + 2 clickable knobs, 3 layers, USB VID/PID `1189:8840`. It replaces the vendor's x86_64 Qt app ("MINI_KEYBOARD"). Swift + IOKit HID only — no hidapi, no third-party dependencies.

## Commands

- Build: `swift build`
- Run: `swift run ch57x-whisperer <command>` — commands: `probe`, `selftest`, `bind`, `led`, `read`, `record`, `gui`, `agent`, `update`
- Verify encoder: `swift run ch57x-whisperer selftest` (checks byte output against ch57x-keyboard-tool test vectors; asserts are active in debug builds only)
- Hardware smoke test: `swift run ch57x-whisperer probe` (keyboard must be connected **by USB/dongle**, not Bluetooth — programming only works over USB)
- Package for release: `scripts/make-dmg.sh` → `dist/CH57x Whisperer.app` + `dist/CH57x Whisperer.dmg` (git-ignored). Builds release, renders the icon via `ICON_PNG` into an `.icns`, assembles the bundle (Info.plist with `CFBundleIdentifier` `com.palanx.ch57x-whisperer` and `CFBundleShortVersionString` copied from the repo Info.plist — bump the version THERE), **nests the faceless agent helper `Contents/Helpers/Action Whisperer.app`** (same binary as `action-whisperer`, its own colored-lips icon via `AGENT_ICON_PNG`, id `com.palanx.ch57x-whisperer.agent`, `LSUIElement`), ad-hoc signs **inside-out** (helper first, then the outer app), packs a drag-to-Applications DMG (one drag installs both; the updater replaces the whole bundle, helper included). No Developer ID: downloaders need right-click → Open once. Releases live on GitHub (`Palanx/CH57x-Whisperer`), uploaded with `gh release create`; the release tag (`vX.Y.Z`) must be newer than shipped versions or the auto-updater won't offer it.

## Architecture

Under `Sources/ch57x-whisperer/`:

- `Protocol.swift` — pure encoder, no I/O. Produces 64-byte wire messages. Ported from [kriomant/ch57x-keyboard-tool](https://github.com/kriomant/ch57x-keyboard-tool) `src/keyboard/k884x.rs`; consult that file for any protocol question before reverse-engineering anything.
- `Device.swift` — IOKit transport. Opens the keyboard's **vendor HID interface** (usage page `0xFF00`; the device also exposes a normal keyboard interface on usage page 1 — never open that one, it triggers Input Monitoring permissions). Sends via `IOHIDDeviceSetReport`: report ID = first byte (0x03), data = remaining 63 bytes.
- `Read.swift` — config read-back (queries + response decoding, see protocol facts below).
- `Record.swift` — listen-only CGEvent tap that records keystrokes as chord tokens (filters auto-repeat).
- `Engine.swift` — host-side action engine (`ch57x-whisperer agent`, roadmap step 5). Menu-bar-only process (`.accessory`; status icon is `mouthIcon()` — the lips from `appIcon()` alone, lower lip squashed to 60%, as a template image so the menu bar tints it; `mouthIcon(update: true)` adds the down-arrow badge when a release is newer; fully standalone, the GUI never needs to be open) registering F13–F20 global hotkeys via Carbon `RegisterEventHotKey`, chosen because it needs **zero permissions** (no Input Monitoring/Accessibility/TCC prompt — the system delivers only the registered keys, so the process can't observe typing; deliberately corporate-Mac-safe — never use the Record.swift event tap here). Each hotkey runs `/bin/zsh` on a script in `~/.config/ch57x-whisperer/actions/`, filename = chord token + `.sh` (`f13.sh`, `cmd-f14.sh`; modifiers ctrl/shift/alt/cmd, keys f13–f20 only — no virtual keycodes exist above kVK_F20), with `FRONT_APP` (frontmost bundle id) and `FRONT_APP_NAME` in the env so one script branches per app; `open -a` is the focus-or-launch primitive. Hotkeys are registered **only for scripts that exist** (a registration consumes the key system-wide); menu offers Reload Scripts / Open Actions Folder / per-script run-on-click. First run writes `f13.sh.example`. **The agent ships as a separate nested app so LaunchServices never confuses it with the GUI**: `CH57x Whisperer.app/Contents/Helpers/Action Whisperer.app`, bundle id `com.palanx.ch57x-whisperer.agent`, `LSUIElement` (no Dock icon ever), same binary copied in as `action-whisperer` (which reports the helper id because a bundled executable's on-disk `Contents/Info.plist` overrides the embedded `__info_plist` — verified). Distinct ids mean opening the GUI can't wake the agent and quitting one can't kill the other; the old v1.2.4 `applicationShouldHandleReopen` GUI-spawn hack is gone. `agent --install`/`--uninstall`: plain path-based `~/Library/LaunchAgents/ch57x-whisperer.agent.plist`; `agentExecutablePath()` points `ProgramArguments` at the nested helper when installing from the main bundle (the GUI), else at the invoking binary (the helper reinstalling itself, or a source build). **Migration** (`migrateToHelperIfNeeded`, runs once at agent startup): installs from ≤1.2.4 have a plist pointing at the main binary — if we're that legacy agent (running from the main bundle with the helper beside us) we rewrite the plist to the helper and reload it *out of band* via a detached zsh script (unloading our own launchd job would SIGTERM us mid-command), then exit so launchd relaunches us as the helper. The GUI exposes agent lifecycle in an "Agent" menu (Start Action Whisperer / Start at Login / Remove from Login → `AgentControl` → `startAgentNow`/`installLaunchAgent`/`uninstallLaunchAgent`). Do NOT switch to `SMAppService` (tried 2026-07-08 for the pretty Login Items icon): it pins the app's code signature (LWCR) and BTM never refreshes the pin for ad-hoc re-signs — after any app update launchd kills the agent with EX_CONFIG/OS_REASON_CODESIGNING, and no unregister/register/lsregister dance fixes it reliably (Apple forums thread 795022 hit the same wall; needs a Developer ID). Adding `AssociatedBundleIdentifiers: [com.palanx.ch57x-whisperer.agent]` to the plain LaunchAgent plist was tried next (2026-07-09, prototyped as v1.3.1) and **reverted** — verified with `sfltool dumpbtm` plus an on-device visual check: BTM does pick up the association, but (a) because the agent is a helper NESTED inside `CH57x Whisperer.app`, BTM rolls it up under its container and shows the name **"CH57x Whisperer"**, never a standalone "Action Whisperer"; and (b) `Developer Name: (null)` (ad-hoc signature) means the helper's colored icon never renders — Login Items keeps a generic icon. A standalone "Action Whisperer" name+icon would need BOTH un-nesting into a second top-level app (giving up the one-drag / one-auto-update bundle) AND a Developer ID for the icon to render. **Decision: accept the generic exec name/icon for the login item; do not re-attempt without a Developer ID.** (The bare `action-whisperer` shown in Finder is a raw executable, not an app bundle — it is always generic regardless.) Logs to `/tmp/ch57x-agent.log`; stdout is line-buffered so the log is live. Relaunch after quit: `launchctl start ch57x-whisperer.agent`. `parseHotkeyName` is covered by `selftest`. Verified end-to-end 2026-07-08 (hotkey → script with env, modifier variants, launchd run) via `osascript` synthetic presses — note scripts spawn async, so give checks ~2 s.
- `GUI.swift` — SwiftUI configurator (`ch57x-whisperer gui`): layer tabs, key/knob grid, chord composer (modifier toggles + searchable key picker, covers F13–F24), colored chip preview of each step (cyan=combination, magenta=named key, orange=media/mouse, red=invalid), media/mouse actions menu, display-only glyphs (`keyGlyphs`/`displayToken`: chips, tiles and picker show punctuation/keypad as the character — `dot`→`.`, `kpdot`→`(kp).`, `shift-kpdot`→`shift-(kp).` — while the binding field, CLI and `read` keep the token names), per-key delay field (0–6000 ms between steps, loaded from `read`, written with the binding; keys with a delay show an orange ⏱N badge next to the tile title), LED section, clickable ⓘ info popovers. The LED preview dot mimics the mode (off=gray, backlight=steady, sweep/sweep-reverse=pulsing since one dot can't show a key-by-key sweep, press=dim static); LED state can't be read from the device, so the LED display is a memory, not a readout: the last value set is remembered per layer in UserDefaults (`led.layer<N>`), shared with the CLI `led` command, which writes the same key. LEDs changed any other way show as *off*. Runs in the CLI binary via NSApplication, no app bundle. Also holds `appIcon()`: the Dock icon (lips whispering over the pad, lower lip tucked behind it) is drawn in code — no asset files; `ICON_PNG=<path> ch57x-whisperer gui` renders a 1024px PNG and exits, which is how `docs/icon.png` (used by README.md) is regenerated after any change to the drawing. `agentIcon()` is the Action Whisperer helper's icon (the same lips full-color and centered, no pad), rendered via `AGENT_ICON_PNG` for make-dmg.sh. The GUI also builds an "Agent" menu (Start / Start at Login / Remove from Login) when run from an `.app`.
- `Update.swift` — auto-update from GitHub releases (`Palanx/CH57x-Whisperer` latest tag vs `CFBundleShortVersionString`). `Updater.shared` checks on launch + every 6 h in both GUI and agent (`host` says which), but ONLY when running from an `.app` bundle — source builds never auto-check (the explicit `update` command works from anywhere); "Restart to Update" downloads the release DMG (floating SwiftUI progress window, errors shown with Close), mounts and replaces `/Applications/CH57x Whisperer.app` **in-process** (so failures still reach the window; running processes keep their inodes), then a detached zsh relauncher kills the recorded pids and restarts exactly what was running — agent via `launchctl start` (path-based plist survives the replace) or, if it wasn't a login item, `open` the nested helper app; GUI via `open -a`; nothing that wasn't running. Process detection quirk: the agent's cmdline ends in ` agent` (the helper's `action-whisperer`, or a source `ch57x-whisperer`), the Finder GUI is the full bundle path — `runningInstances()` greps `pgrep -fl whisperer` (catches both binary names), matches by shape, kills by pid, never by pattern. `ch57x-whisperer update` = same machinery from the CLI (used for e2e testing: set repo Info.plist version low, force relink, run it). `isNewerVersion` is covered by selftest.
- `Info.plist` (repo root) — embedded into the binary via linker flags in `Package.swift` (`__TEXT,__info_plist`) so the menu bar shows "CH57x Whisperer" instead of the process name. Has `CFBundleIdentifier` `com.palanx.ch57x-whisperer` (added 2026-07-08): this moved the UserDefaults domain from the old process-name one — main.swift migrates the `led.layer<N>` keys once at startup; keep the id in sync with make-dmg.sh's bundle plist. Also holds `CFBundleShortVersionString`, the single version source. CAUTION: swift build does NOT track this file — force a relink (`rm .build/*/debug/ch57x-whisperer`) after editing it.
- `main.swift` — CLI parsing and dispatch. Also holds the undocumented `raw` command (absent from `usage()` on purpose): `raw 03 fe b0 01 08 ... -- 03 fd fe ff` sends arbitrary hex messages, `--`-separated, each zero-padded to 65 bytes and all sent through **one** open session (which is what let us reproduce the vendor's commit sequence). Protocol lab bench — it validates nothing. No args = run the agent when the bundle id is the helper's (`com.palanx.ch57x-whisperer.agent`), else open the GUI when running from an `.app` bundle (Finder double-click), else `probe` from a terminal. Token tables (`keyCodes`, `mediaCodes`, `mouseButtons`) and `bindMessages` are shared with the GUI, so `read` output is always valid bind input. Spanish ISO aliases (`ñ`, `ç`) parse to the matching HID codes; unnamed codes print/parse as `0xNN`.

## Protocol facts (CH57x 884x variant)

- Every message: 64 bytes zero-padded, starts `0x03`.
- Bind: `[0x03, 0xfe, keyID, layer, 0x01, 0,0,0,0,0, count, (mod, code)...]` then finish sequence `[03 aa aa]`, `[03 fd fe ff]`, `[03 aa aa]`.
- Layer wire byte is 1–3 (not 0-based). Key IDs: buttons 1–12; knob N (1-based) is `16 + 3(N-1) + action` with ccw=0, press=1, cw=2.
- Modifier bitmask: ctrl=1, shift=2, alt=4, cmd=8. Key codes are standard USB HID usages (a=0x04, f13=0x68).
- Max 18 chords per key. Delay: ONE value per key (message type `0x05`, u16 LE ms, max 6000), applied between every step of the sequence — not positional, keyboard chords only. CLI: `bind ... --delay <ms>`. Whether firmware also pauses before the first stroke is untested.
- Media bind: type byte `0x02`, consumer usage u16 LE at bytes 11–12 (playpause=0xCD, calculator=0x192). Mouse bind: type `0x03`, subtype at byte 10 (1=click, 3=wheel), modifiers at 11, buttons at 12 (left=1 right=2 middle=4), wheel delta at 15 (1=up, 0xFF=down). Media/mouse bindings hold ONE action, never a chord sequence.
- LED: `[0x03, 0xfe, 0xb0, layer, 0x08, 0,0,0,0,0, 0x01, 0, (color<<4)|mode]` + `[03 fd fe ff]`, preceded by the commit preamble below. Colors: 0=white 1=red 2=orange 3=yellow 4=green 5=cyan 6=blue 7=purple. LED state is per layer and only visible on the layer the physical switch is on; it can never be read back. The LEDs are also dark right after connecting — they light up on the first layer-switch change.
- Modes, verified on hardware 2026-07-20 **with correct 65-byte messages**: 0=off, 1=steady backlight, 2=sweep top-left → bottom-right (`sweep`), 3=the mirrored sweep (`sweep-reverse`), 4=lights the key you press, 5=always white. The reference ports call 2/3 `shock`/`shock2`; we renamed them to what they do.
- The "breathing" behaviour these modes used to show was **an artifact of the truncated 64-byte messages**, not a real firmware mode — it disappeared once the wire length was fixed. By the same token, the earlier "mode nibbles 6–15 are no-ops" finding was measured with the broken messages and is **not trustworthy**; 6–15 have never been swept properly and may hold more modes.
- **LED persistence — SOLVED 2026-07-20.** LED colours used to die on every keyboard power cycle (they applied live, then reverted). Two causes, neither of them in the LED message's content, which was correct all along:
  1. **Wire length.** Messages must be 65 bytes (report ID + 64 payload); we sent 64. `Read.swift`'s query constants were genuine 65-byte captures that we truncated with `.prefix(64)`.
  2. **The commit preamble.** The write only reaches flash when preceded by the vendor's query sequence — `03 fb fb fb 01` then `03 fa 0f 03 <layer> 7f` for layers 1,2,3 — **on the same open HID session**. `Ch57x.ledCommitPreamble()`; `setLED` prepends it, so always send the whole returned array through ONE `KeyboardDevice`. Neither reference port does this, which is why upstream has the same bug.
  Losing either is silent (the LED still lights up, it just stops persisting), so `selftest` pins the 65-byte length and the preamble's shape.
- How it was found: the vendor app (`~/Downloads/LEN/MINI_KEYBOARD.app`, x86 Qt under Rosetta) is **unsigned**, so `DYLD_INSERT_LIBRARIES` with a small dylib interposing `IOHIDDeviceSetReport` logs exactly what it sends — no Wireshark, no sudo, no VM. That capture answered in one shot what ~8 power cycles of blind sweeping could not. **If a protocol question ever comes up again, capture the vendor app first.** Note it holds the HID device exclusively: while it is open our writes fail with `0xe00002c5`.
- Ruled out along the way, so nobody repeats it: the `[03 aa aa]` bookends are not a commit; mode nibbles 6–15 are no-ops (this closes the 2026-07-07 "observed behavior undocumented" question); `read` after a write does not commit. Caveat on all of it: the first batch appears to have *erased* layer 1's slot, after which the "did flash change?" signal was gone and every later bisection had no detection power — those no-ops proved nothing at the time.
- Still open: the vendor exposes a **key-by-key sweep mode** (top-left → bottom-right, plus its mirror) that no value of our mode nibble reaches, so its LED encoding differs from ours somewhere. Capture it the same way rather than guessing.
- Read-back (see `Read.swift`, ported from [kamaaina/macropad_tool](https://github.com/kamaaina/macropad_tool)): send device-type query `0x03 0xfb...`, then per-layer `0x03 0xfa keys knobs layer...`; device answers one input report per key/knob action shaped like the bind message (delay is big-endian in responses). Query tails are verbatim USB captures — replay them exactly.

## macOS permissions

- `ch57x-whisperer record` needs **Input Monitoring** (System Settings > Privacy & Security > Input Monitoring; open it directly with `open "x-apple.systempreferences:com.apple.preference.security?Privacy_ListenEvent"`). The first failed run adds the terminal to the list; after enabling the toggle, the terminal must be **fully restarted** for the grant to take effect.
- `screencapture` (used to screenshot the GUI for verification) needs **Screen Recording** for the terminal — same rule: grant the toggle, then fully restart the terminal.
- Talking to the config interface (usage page `0xFF00`) needs no permission — only event taps and screen capture do.
- `ch57x-whisperer agent` needs **no permission at all** — `RegisterEventHotKey` is TCC-free by design. Keep it that way: adding anything Accessibility/Input-Monitoring-shaped to the agent defeats its reason to exist (runs on a managed corporate Mac).

## Hardware constraints

- macOS has no virtual keycodes for F21–F24 (`Events.h` ends at `kVK_F20`), so apps never see those keys — but the HID events DO arrive (verified on this machine: `hidutil` UserKeyMapping remap of F24 works). So F21–F24 are usable via hidutil remap or by reading the raw HID device; hidutil remaps reset on reboot.
- Bluetooth mode drops extended F-keys; anything clever must assume the dongle/USB connection.
- Bindings persist in the keyboard (it has a battery); a bad write only costs re-binding a key, not a brick.
- Disconnected keyboard is handled gracefully everywhere (verified): CLI commands print `error: keyboard not found — is it connected by USB?`, the GUI shows the same in its status line and keeps running.
